## Design decisions

- ERCOT zone → PyPSA 5-bus mapping: COAS/EAST→HOUSTON, NCEN/NRTH→NORTH, FWES/WEST→WEST, SCEN/SOUT→SOUTH (PAN has no weather-zone equivalent). The UI groups these into a "Hub" selector (HB_HOUSTON/HB_NORTH/HB_WEST/HB_SOUTH) matching the naming used on the ERCOT Historical tab, so the whole platform speaks one zone language.
- **Do not** back-calculate system-wide load by dividing one hub's stressed load by its `LOAD_FRACTIONS` share (`network.py` `_T1_LOAD`) — those synthetic Tier-1 fractions don't match real EIA-930 zone shares and overstate system load by ~30%+. Instead fetch daily rows for **all 8** weather zones (`GET /api/load-forecast/zones`, no filter) and sum them per calendar day to get the true system-wide coincident peak; feed that real total into `system_load_mw` — the backend OPF already redistributes it across buses internally via its own fractions. Only use per-hub aggregation for the hub-specific chart/peak display, never for the value sent to `/pypsa/scarcity`.
- Peak-day selection: pick the single calendar day with the highest total (`baseMw+evMw+dcMw` at 100% scenario) across the whole 3-year forecast, then apply the EV%/DC% sliders to that day's real base/ev/dc split. Don't use a monthly-average "peak" — daily data is available and materially more accurate than an averaged month.
- Slider redesign pattern for scenario/stress-test controls: show the real absolute value (MW/GW) prominently above the slider (where a bare % used to sit) and keep the %/multiplier reading small below it, with a one-line caption. Calibrate default = real observed average, ceilings/max ≈ 2σ beyond observed best/worst case where variance data exists (e.g. ERCOT wind/solar CF from `ercot_fuel_mix`, gas forced-outage rate from `thermal_params`); where no variance data exists (EV/DC adoption growth), label the range as a "scenario range" instead of a fabricated statistical bound, and say so in a methodology note.
- `/pypsa/scarcity` returns HTTP 422 `{detail: "..."}` when load exceeds all available capacity (fully infeasible LP) rather than a 200 with a result body. The existing `pypsa-scarcity.tsx` reference page does NOT handle this (silently falls back to a fake "NORMAL" display) — new pages should check `res.ok` and surface the 422 detail as an explicit "CRITICAL — Infeasible" state instead of copying that fallback pattern.
- `total_available_mw` in the `/pypsa/scarcity` response is not full dispatchable capacity (it can read far below `system_load_mw` even when the OPF dispatches the full load with `status: "optimal"` and zero shed) — don't treat it as a capacity-adequacy check on its own; trust `scarcity_level`/`total_load_shed_mw` instead.
- CAISO has no load forecast dataset and no nodal OPF model. Approach used: real EIA-860 installed capacity (`GET /api/caiso-capacity?hub=SP15|NP15`, aggregated from `candidatesTable` where market=CAISO) × capacity-factor/derate assumptions per fuel type, compared against a user-specified load slider (no forecast to anchor it), computed reactively client-side (no backend round-trip needed) — explicitly labeled as a lower-rigor "reserve-margin estimate" vs ERCOT's full nodal OPF.
- `candidates` schema exports as `candidatesTable` (not `candidates`) from `@workspace/db/schema` — easy to get wrong when writing a new aggregation route.
